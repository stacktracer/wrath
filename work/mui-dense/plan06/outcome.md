# MUI Dense, Plan06 Outcome

The work landed as a local dense-library directory at `src/mui-dense/dense/`. The gallery now consumes that directory's public surface rather than wiring the internals together itself: presets, the top-level `createDenseTheme` builder, and `DenseDataGrid`. `app.tsx` no longer coordinates theme assembly by hand. Instead it keeps gallery-local state, adapts that state through `src/mui-dense/gallery-density.ts`, and calls the same top-level theme builder that a real consumer would call. Gallery-only labels, explanatory copy, layout-scale/image-gap controls, and other page-shaped metadata no longer live in the reusable layer.

Data Grid was the only wrapper that graduated in this pass. `DenseDataGrid` now owns row and header measurement, keeps the probe DOM internal, forwards ordinary `DataGridPro` props, and exposes a small app-facing `cellBlockPadding` knob in either `px` or `ex`. The gallery uses that surface directly and shows the measured row/header heights through the wrapper's metrics callback. No broader wrapper set was added. The rest of the dense policy still rides through the theme builder and ordinary MUI props, which matches the current code pressure better than introducing a wider wrapper family prematurely.

The direct tests were retargeted to the new public surface rather than the deleted internal modules, and the page-level browser test still proves the gallery works as a consumer. One implementation-shaped choice is worth keeping in mind for the next round: the new Data Grid tuning knob currently adjusts block padding and the measured heights together, not full four-sided cell padding. That was enough to hide the measurement plumbing and satisfy the current compactness use case, but future app usage may show pressure for a wider horizontal-spacing control as well.
