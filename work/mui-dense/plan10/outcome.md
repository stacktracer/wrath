# MUI Dense, Plan10 Outcome

This pass changed the reusable `DenseDataGrid` padding surface from a structured `{ unit, value }` object to a plain CSS length string. `DenseSettings`, `DenseDataGridOptions`, presets, and the gallery adapter now all carry `cellBlockPadding` as text such as `1px` or `0.5cap`, and I removed the old `DenseDataGridCellBlockPadding` type instead of preserving a wrapper that only serialized back to CSS anyway.

For sizing, the grid now asks the browser to do both validation and unit resolution. At render time it normalizes the supplied string, uses `CSS.supports('padding-top', value)` to reject unsupported values, and falls back to the default `1px` padding when the input is empty or invalid. During measurement it applies the resolved padding string to a hidden probe that inherits `body2` typography, then reads the computed top padding in pixels. That keeps `cap`, `ex`, `em`, `rem`, and similar browser-supported units in the browser's hands instead of reimplementing them in TypeScript. The row/header height math still happens in one shared helper after the probe values are read.

The gallery layer changed shape as well. It no longer splits padding into numeric value and unit fields, and the sidebar no longer uses the old slider/select control model. The Data Grid card now shows the current CSS length, offers quick example buttons for `1px` and `0.5cap`, and exposes a text field for arbitrary CSS length input. I kept the default preset behavior at `1px`, but switched the text-relative example path and tests to `0.5cap`.

One subtle implementation detail was important here: font-relative padding units need to resolve in the same font context as the grid text. The hidden padding probe therefore inherits `theme.typography.body2` before the browser resolves the supplied CSS length. Without that, `cap` and `ex` would have been measured against the browser default font instead of the dense grid typography.

This pass also turned up an unresolved behavior bug that should be handled separately: when `cellBlockPadding` is pushed to large values such as `20px`, `2cap`, or `3ex`, Data Grid text and header icons clip badly even though the wrapper computes a larger row and header height. The likely cause is that the padding is still applied to the fixed-height outer `.MuiDataGrid-cell` and `.MuiDataGrid-columnHeader` slots, whose content is already governed by MUI X height, line-height, and overflow rules. That bug is related to the padding feature, but it is a follow-up to how the padding is applied inside the grid, not to the CSS-length-string API itself.
