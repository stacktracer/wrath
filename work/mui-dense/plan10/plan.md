# MUI Dense, Plan10

`DenseDataGrid` currently exposes `cellBlockPadding` as `{ unit: 'px' | 'ex', value: number }`, but the reusable layer and the DOM both really want a CSS length string. The structured type forces the library to special-case units in TypeScript, leaks the gallery's current control model into the reusable API, and makes it awkward to use better text-relative units such as `cap`.

Change the reusable `DenseDataGrid` surface so `cellBlockPadding` is a CSS length string. Prefer a plain string unless a lightweight alias would improve clarity substantially. Do not keep `DenseDataGridCellBlockPadding` as a faux abstraction if it only serializes back to CSS text. For metric calculation, resolve the length by asking the browser, not by reimplementing unit conversion in TypeScript. The sizing path should use a hidden probe whose size or padding is set from the supplied CSS length so browser-supported length units such as `px`, `cap`, `ex`, `em`, and `rem` all resolve naturally. Guard the metric math against invalid or unresolved values by falling back to the default dense padding instead of letting `NaN` leak into row and header sizes.

After improving the reusable API and impl, update the local dense/gallery layer to match. `DenseSettings` and presets should carry a string, the gallery adapter should stop splitting padding into value and unit fields, and the sidebar should use a local control model that fits the new string API rather than preserving the old `px`/`ex` slider shape by inertia. Use `cap` for the text-relative example and control path unless execution turns up a concrete reason not to.

Update the dense-data-grid tests to cover both the default `1px` behavior and a text-relative `cap` example, and update any README text that still describes the older structured padding API. Record in `outcome.md` how the browser-based length resolution landed, especially if the final implementation needs any constraints or fallbacks beyond "accept a CSS length string."
