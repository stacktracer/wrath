# MUI Dense, Plan09 Outcome

This pass cleaned up `src/mui-dense/lib/data-grid.tsx` without changing the public `DenseDataGrid` API. I kept the exported `DenseDataGridMetrics` name and removed the internal `DenseDataGridMeasurement` type instead of trying to invent a second public-facing metrics-adjacent name. The internal code now uses clearer local names such as `gridRoot`, `body2TextProbe`, and `xHeightProbe`, and the animation-frame path reads the probes directly before building the published metrics.

I kept a small pure `createDenseDataGridMetrics()` helper because the fallback path and the live DOM-measurement path still share the same sizing math, but I did not keep a second helper just for DOM measurement. That left the flow easier to read: measure the probes, create the published metrics, compare against current metrics, and update state only when the published values materially changed. I also pulled the equality check into its own helper and added the requested multi-line comment directly on the probe-reading block, describing what the probes measure, why the final height calculation combines text, x-height, checkbox size, and block padding, and where the implementation is still a bit fragile.

I kept the functional `setMetrics(currentMetrics => ...)` form. The measurement happens in `requestAnimationFrame()`, so by the time that callback runs the component may already have newer state than the render that scheduled the frame. Comparing against the live state value inside the setter is the clearest way to avoid replacing newer metrics with an older frame's result. I did not expose a new pure helper just to add unit tests; the existing browser-level `DenseDataGrid` tests remained the main protection for this refactor.
