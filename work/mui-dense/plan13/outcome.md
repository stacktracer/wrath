# MUI Dense, Plan13 Outcome

This pass moved Data Grid sizing to a cleaner boundary. `src/mui-dense/lib/data-grid.tsx` no longer has `DenseDataGrid` infer `body2` for itself. Instead, `lib` now exposes a layered API: a pure sizing core (`createDenseDataGridMetrics` plus `createDenseDataGridHeightProps`), a lower-level DOM line-height helper (`measureDenseDataGridLineHeights`), and a `DenseDataGrid` wrapper that either measures its own representative probes or consumes precomputed metrics when a caller supplies them.

The gallery now uses that same layered surface directly through `DenseDataGrid` itself. The wrapper owns the hidden probes, reads typography from the live grid root through a direct ref rather than a sibling search, and reports the measured metrics back to the sidebar with `onMetricsChange`. Callers can still override the representative row/header probe content when they need to, or skip probing entirely by passing explicit metrics. The direct browser test was updated to use that default internal probe path for the normal case, and a new pure test file locks down the fallback line-height rules, device-pixel snapping, and density-factor compensation. The browser suite also now proves that the wrapper can consume separate row and header metrics without another API rewrite.

One boundary remains intentionally asymmetric. The probe path resolves CSS-length padding accurately because it measures hidden probes in the live page, so units like `cap`, `em`, and `rem` work there as intended. The no-probe fallback path stays conservative: it can create default metrics from fallback line heights, but it only resolves pixel padding exactly and otherwise falls back to the default `1px` value. That tradeoff felt acceptable because the accurate path is now the default inside `DenseDataGrid`, while explicit metrics are still available when an application wants tighter control.
