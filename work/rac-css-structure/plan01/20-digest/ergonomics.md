# Digest: Developer Ergonomics And Cognitive Load

## Recommendations

- Prefer a small, named semantic layer over a large vocabulary of component-specific knobs. The strongest examples here are [Open Props](/home/mike/Documents/projects/wrath/code/work/rac-css-structure/plan01/10-research/open-props.md) for shallow token packs and [Radix Themes](/home/mike/Documents/projects/wrath/code/work/rac-css-structure/plan01/10-research/radix-themes.md) for a compact theme surface. The main thing to preserve is learnability: a developer should not need to memorize dozens of unrelated token families before making a safe change.
- Keep styling in plain CSS with direct `data-*` selectors, and do not move the center of gravity to utility strings or prop-driven styling. [Tailwind](/home/mike/Documents/projects/wrath/code/work/rac-css-structure/plan01/10-research/tailwind.md) and [Primer](/home/mike/Documents/projects/wrath/code/work/rac-css-structure/plan01/10-research/primer.md) both show that powerful systems can still impose a large syntax burden on day-to-day authors.
- Make the common path obvious and the escape hatch explicit. [React Spectrum](/home/mike/Documents/projects/wrath/code/work/rac-css-structure/plan01/10-research/react-spectrum.md) is the cautionary example: the safe path is easy, but the moment a team needs real visual ownership the mental model gets steep. We want the opposite balance, with local CSS ownership as the normal path and narrow, documented overrides as the exception.
- Favor a small number of global density and spacing knobs that can flow through many components. The notes on [Radix Themes](/home/mike/Documents/projects/wrath/code/work/rac-css-structure/plan01/10-research/radix-themes.md), [Primer](/home/mike/Documents/projects/wrath/code/work/rac-css-structure/plan01/10-research/primer.md), and [Open Props](/home/mike/Documents/projects/wrath/code/work/rac-css-structure/plan01/10-research/open-props.md) all suggest that developers can reason about compactness more easily when a single density axis feeds multiple surfaces.
- Use component-local tokens only when a component truly needs private tuning. This keeps the global vocabulary small and prevents every exception from becoming a new concept. [Primer](/home/mike/Documents/projects/wrath/code/work/rac-css-structure/plan01/10-research/primer.md) and [Radix Themes](/home/mike/Documents/projects/wrath/code/work/rac-css-structure/plan01/10-research/radix-themes.md) both show the value of a clear boundary between shared semantic values and component-specific adjustments.
- Treat theme switching as a document-level concern, not a per-component concern. [Tailwind](/home/mike/Documents/projects/wrath/code/work/rac-css-structure/plan01/10-research/tailwind.md), [Open Props](/home/mike/Documents/projects/wrath/code/work/rac-css-structure/plan01/10-research/open-props.md), and [Primer](/home/mike/Documents/projects/wrath/code/work/rac-css-structure/plan01/10-research/primer.md) all work best when the mode selection is consistent and predictable, while [React Spectrum](/home/mike/Documents/projects/wrath/code/work/rac-css-structure/plan01/10-research/react-spectrum.md) and [Radix Themes](/home/mike/Documents/projects/wrath/code/work/rac-css-structure/plan01/10-research/radix-themes.md) become heavier when theme decisions are hidden behind a framework API.
- Keep the learning surface centered on CSS variables, selectors, and a small set of semantic names. Avoid introducing extra concepts like utility prefixes, prop taxonomies, or framework-specific styling macros unless they earn their keep immediately.

## Open Questions

- How small can the semantic token vocabulary be before it stops covering the real variation we need across RAC components?
- Which adjustments should be global knobs, and which should remain component-local overrides?
- Do we want a single density axis such as `compact` to drive most spacing changes, or a few separate knobs for spacing, radius, and typography?
- How much local escape power should be exposed directly in author-facing CSS before the model becomes too easy to fragment?
- Should component-local tokens be encouraged early, or reserved for the small set of cases where shared semantic tokens cannot express the need?
- Do we need a formal theme mechanism beyond selector-scoped variables, or can a simple data-attribute theme switch cover the use cases without extra cognitive load?
- What is the minimum documentation surface that still lets a mixed-experience team make safe styling changes without reading the whole token system?
